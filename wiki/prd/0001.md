# PRD-0001: 적응형 파이프라인 (Adaptive Pipeline)

> Source: REQ-001
> Date: 2026-03-10
> Status: Draft

## 1. 배경

현재 `/teams` 파이프라인은 모든 작업에 동일하게 4개의 서브에이전트를 사용한다. 작은 작업(설정값 변경, 유틸 함수 추가 등)에서도 서브에이전트를 스폰하면 컨텍스트 초기화 + API 호출 오버헤드가 크다. 반면, 서브에이전트를 쓰지 않으면 TDD 사이클에서 컨텍스트가 폭발하여 이후 단계가 망가진다.

## 2. 목표

- 작업 규모에 따라 서브에이전트 사용을 자동으로 조절하여 오버헤드를 최소화한다
- TDD 사이클의 안정성은 유지한다 (Medium 이상에서 서브에이전트 분리)
- 기존 스킬 파일은 변경하지 않고, 파이프라인 프롬프트(teams.ts)만 수정한다

## 3. 범위

### In Scope

- `teams.ts` 프롬프트에 작업 크기 판단 로직 추가
- `teamsEn.ts` 영문 버전 동기화
- Small/Medium/Large 3단계 모드 정의

### Out of Scope

- 스킬 파일(devImpl.ts, testImpl.ts, finalize.ts 등) 수정
- commandQueueService.ts 등 런타임 코드 변경
- bugfix-teams 파이프라인 변경 (별도 요구사항으로 분리)

## 4. 기능 요구사항

### FR-1: 작업 크기 판단

Step 1(req-manage/PRD 작성) 완료 후, 메인 에이전트가 PRD 내용을 읽어 다음 기준으로 모드를 결정한다:

| 기준 | Small | Medium | Large |
|------|-------|--------|-------|
| 파일 변경 수 | 3개 이하 | 4~10개 | 10개 초과 |
| 새 모듈 수 | 0개 | 1~2개 | 3개 이상 |
| UI 포함 여부 | 없음 | 무관 | 있으면 자동 Large |

여러 기준 중 하나라도 상위 모드에 해당하면 상위 모드를 적용한다.

### FR-2: Step 1 인라인화

모든 모드에서 Step 1(req-manage)은 메인 에이전트가 직접 실행한다. PRD 작성은 파일 읽기 + 문서 작성으로 항상 가볍다.

### FR-3: 모드별 파이프라인 실행

| 모드 | Step 1 | Step 2 | Step 3 | Step 4 |
|------|--------|--------|--------|--------|
| Small | 메인 직접 | 메인 직접 | 메인 직접 | 메인 직접 |
| Medium | 메인 직접 | 메인 직접 | 서브에이전트 | 메인 직접 |
| Large | 메인 직접 | 서브에이전트 | 서브에이전트 | 서브에이전트 |

### FR-4: 검증 규칙 유지

모드에 관계없이 각 단계의 검증 기준은 동일하다:
- Step 1: wiki/prd/{NNNN}.md 존재 (>100B)
- Step 2: wiki/specs/{NNNN}.md 존재 (>100B), UI면 wiki/mockups/ HTML
- Step 3: wiki/tests/{NNNN}.md + wiki/tdd/{NNNN}.md 존재 (>100B) + 테스트 통과
- Step 4: wiki/deploy/{NNNN}.md 존재 (>100B) + wiki/views/index.html

### FR-5: 모드 출력

파이프라인 시작 시 결정된 모드를 사용자에게 출력한다:
```
[Pipeline Mode: Small] 서브에이전트 0개 — 메인 에이전트 직접 실행
```

## 5. 비기능 요구사항

- 프롬프트 변경만으로 구현 (런타임 코드 변경 없음)
- 기존 파이프라인 동작과 역방향 호환 (Large 모드 = 현재 동작)

## 6. 성공 기준

- Small 작업에서 서브에이전트 호출 0회로 파이프라인 완료
- Medium 작업에서 서브에이전트 1회(TDD)로 파이프라인 완료
- Large 작업에서 기존과 동일하게 3회 서브에이전트로 파이프라인 완료
- 모든 모드에서 산출물 검증 통과
